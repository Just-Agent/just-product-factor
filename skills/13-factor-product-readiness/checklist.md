# 13-factor-product-readiness Checklist

## 1. Product promise

- [ ] The README explains what the product does in one sentence.
- [ ] The target user and primary use case are explicit.
- [ ] The slogan is memorable and consistent.

## 2. First 30 seconds

- [ ] A visitor can understand the value without reading the whole repository.
- [ ] The README has a clear feature list and use-case table.
- [ ] The repository avoids vague demo-only language.

## 3. First 5 minutes

- [ ] Setup commands are copyable.
- [ ] A minimal example runs without hidden assumptions.
- [ ] Required environment variables and dependencies are documented.

## 4. Examples and recipes

- [ ] Examples map to real user workflows.
- [ ] Recipes are small, complete, and copyable.
- [ ] Example outputs or expected results are shown.

## 5. Product surface

- [ ] CLI/API/UI names are consistent.
- [ ] Defaults are useful.
- [ ] Output paths, filenames, and config conventions are predictable.

## 6. Failure recovery

- [ ] Error messages explain what happened and what to do next.
- [ ] Logs are human-readable.
- [ ] Common failure cases have troubleshooting notes.

## 7. Documentation structure

- [ ] README is not overloaded with every detail.
- [ ] Guides, reference docs, and examples are separated.
- [ ] Docs include both quickstart and advanced usage.

## 8. Validation path

- [ ] Build, test, lint, type check, or smoke test commands are documented.
- [ ] CI runs meaningful checks.
- [ ] Validation limits are documented honestly.

## 9. Release readiness

- [ ] VERSION, CHANGELOG, RELEASE_NOTES, LICENSE, CONTRIBUTING, and ROADMAP exist.
- [ ] Versioning rules are clear.
- [ ] Release artifacts are reproducible.

## 10. GitHub polish

- [ ] The repository has helpful issue/PR templates or contribution guidance.
- [ ] The project has a clear roadmap and next milestones.
- [ ] Badges, links, and status notes are accurate.

## 11. Trust and safety

- [ ] Security or privacy assumptions are documented where relevant.
- [ ] Risky defaults are avoided.
- [ ] External services, tokens, and credentials are handled clearly.

## 12. Agent-native path

- [ ] The repository explains how an Agent should use, modify, or validate it.
- [ ] Prompts, templates, or machine-readable manifests exist when helpful.
- [ ] Repeatable optimization loops are documented.

## 13. Retention and differentiation

- [ ] The project explains why it is better than ad-hoc scripts or generic templates.
- [ ] The user has a reason to return after first use.
- [ ] The repository has a recognizable product identity.
